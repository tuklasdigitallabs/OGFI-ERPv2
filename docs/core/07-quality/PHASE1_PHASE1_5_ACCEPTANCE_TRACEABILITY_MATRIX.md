# Phase I And Phase 1.5 Acceptance Traceability Matrix

**Status:** Implementation evidence mapped; UAT execution pending  
**Created:** 30 June 2026  
**Related records:** `TEST_STRATEGY_AND_UAT_PLAN.md`, `PHASE1_PHASE1_5_UAT_EVIDENCE_PACK.md`, `PHASE1_PHASE1_5_UAT_EXECUTION_SCRIPTS.md`

## Purpose

This matrix ties each Phase I and Phase 1.5 acceptance workflow to the automated checks that currently support it and the manual UAT evidence still required before GO.

Automated tests prove deterministic service, permission, validation, navigation, reporting, and boundary rules. They do not replace pilot UAT. A workflow is release-accepted only when the matching UAT scenario in `PHASE1_PHASE1_5_UAT_EVIDENCE_PACK.md` has execution date, tester, evidence reference, pass result or approved waiver, defect disposition, and signoff.

## Status Key

| Status | Meaning |
|---|---|
| Automated support present | Existing tests cover important deterministic controls for the workflow. |
| Manual UAT required | The workflow still needs staged or pilot execution with screenshots, record IDs, exports, audit logs, or denial proof. |
| External release gate | The workflow depends on deployment, rollback, training, or management signoff outside automated tests. |

## Automation Gaps To Track

These gaps do not invalidate the current service-level baseline, but they must be visible during GO/NO-GO review:

- Full pilot-browser and real-device mobile/tablet UAT evidence is still manual for the operational workflows and project tracker workflows in this matrix. The local Playwright desktop/mobile e2e smoke on 30 June 2026 passed 10 tests, but it does not replace signed UAT evidence.
- `pnpm test:access-control` is intentionally separate from the standard `pnpm test` command because it needs a migrated PostgreSQL database. CI provisions PostgreSQL and runs the named gate; local release-candidate verification must also run it with a valid `DATABASE_URL`.
- Notification and date behavior now has direct helper coverage in `apps/web/src/server/services/notifications.test.ts`, `apps/web/src/server/services/projectNotifications.test.ts`, and `apps/web/src/server/services/projectDates.test.ts`, but full browser notification delivery, manual approval/project scan execution, and mobile/tablet due-date behavior still require UAT evidence.
- Master-data services now have focused coverage in `apps/web/src/server/services/items.test.ts` and `apps/web/src/server/services/suppliers.test.ts`, but pilot master-data readiness still needs manual signoff because item/supplier/UOM quality depends on business-owned data.
- Deferred controls in the UAT evidence pack, including full post-receiving PO amendment and supplier/location/line-add/delete/substitution/payment-term amendment, backdated operational correction, partial receiving-line reversal, transfer dispatch reversal, automated replacement/finance settlement, binary project attachment upload/download, and automated notification scheduling, must stay documented as exceptions unless explicitly implemented and retested.

## Acceptance Rule

No workflow in this matrix is complete from automated tests alone. A workflow reaches release acceptance only when:

1. referenced automated checks pass for the release candidate;
2. the matching UAT scenario is executed with evidence;
3. permission denial and invalid-transition behavior are proven where applicable;
4. audit, source-record, and ledger effects are captured where applicable;
5. blocker/critical defects are fixed or formally waived with mitigation; and
6. the accountable business owner signs off.
