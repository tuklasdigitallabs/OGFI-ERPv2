# Phase 3 Controlled Foundation Acceptance Matrix

**Status:** Controlled foundation acceptance checklist  
**Purpose:** Prove what can be demonstrated safely before production finance, settlement, binary evidence upload, budget hard-block rollout, and workforce transfer go-live gates are completed.

This matrix separates foundation acceptance from production go-live acceptance. Passing a foundation scenario proves the current ERP workflow is controlled, auditable, scoped, and non-mutating where required. It does not approve official accounting production use.

## Acceptance Rules

- Every scenario must preserve tenant, company, brand, location, department, requester/owner, status, and audit context where applicable.
- Approval scenarios must prove no-self-approval and direct URL scope denial.
- Evidence-reference, attachment-metadata-readiness, private binary upload/download, and configured evidence-storage policy scenarios may pass for foundation demos, but production storage-provider approval, scan or scan-waiver approval, recovery proof, and UAT evidence remain blocked by `P3-BLOCK-002`.
- Finance scenarios must not mutate procurement, receiving, inventory, bank balances, AP settlement, or journals unless the specific workflow is explicitly a controlled finance/journal workflow.
- Export/report scenarios must be read-only and preserve scope filters.
- UI/UX readiness is part of foundation acceptance for client-facing scenarios. A scenario cannot pass as client-presentable if the workspace is a long stacked-card page, lacks tabs/subworkspace navigation for multiple jobs, lacks pagination beyond 10 possible records, repeats large inline action forms, or fails readable light/dark contrast.

## Manual Evidence Required Per Scenario

Each executed scenario must capture:

1. User and role used.
2. Company, brand, and location context.
3. Source record reference before action.
4. Action result and next status.
5. Denied action or invalid transition proof where applicable.
6. Audit event or activity history proof.
7. Export/report proof where relevant.
8. Open blocker reference if the scenario stops at a foundation boundary.
9. For client-facing scenarios, screenshot or screen-record proof that the workspace uses the approved list/tab/detail/action pattern and does not present a long stacked-card workspace.

## Deferred Production Gates

The deferred production gates are tracked in `docs/core/07-quality/PHASE3_DEFERRED_GO_LIVE_BLOCKERS_FOR_UAT.md`. No Phase 3 scenario may be marked production-ready while its linked blocker remains unresolved or unwaived.
