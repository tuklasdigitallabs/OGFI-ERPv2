# Creating A Project From A Template

**Audience / required role:** Project Managers and Project Administrators with project create access  
**Applies to:** Project creation using published templates  
**Related phase/module:** Phase 1.5 / Project Templates and Projects  
**Last verified against:** implemented published-template list, project creation, template snapshot, starter tasks, checklist items, milestones, project activity, and scope checks

## Purpose

Use this article to create a scoped project from a published template.

Templates provide starter statuses, tasks, checklist items, and milestones. Applying a template copies a snapshot into the new project, so later template edits do not silently rewrite active projects.

## Before You Start

- Your role must include project create access.
- A published project template must exist.
- You must choose a valid project scope within your authorized company, brand, or location access.
- Know the project sponsor, project manager, target dates, project type, and whether visibility should be restricted.

## Navigation Path

`Projects`

Administrators configure templates in:

`Project Templates`

## Steps

1. Open `Projects`.
2. In `Create Project`, choose a published template.
3. Enter project name, code, project type, description, sponsor, manager, and target dates where shown.
4. Choose the project scope.
5. Mark the project restricted only when explicit membership should be required.
6. Create the project.
7. Review the Project Registry, Project Health, Members, starter tasks, and milestones.
8. Add members before assigning work.
9. Activate the project when it is ready to execute.

## Expected Result

- The project is created in the selected scope.
- Starter tasks, checklist items, and milestones are copied from the template.
- Project activity records the creation/template application.
- Restricted projects are visible only to authorized users.

## Important Controls And Warnings

- Templates affect future projects only after publication.
- Creating a project does not create PRs, POs, receiving records, transfers, inventory movements, or approvals.
- Restricted visibility should not be used casually; it changes who can discover the project.
- Project source-record links can be added later but remain read-only references.
