# OGFI ERP Glossary

| Term                                  | Meaning                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Company                               | A legal or operating entity using the ERP, such as One Gourmet Foods Inc.                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| Organization code                     | The tenant identifier entered with an email and password to sign in to the correct ERP organization.                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| Brand                                 | A restaurant concept operated under a company.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| Location                              | A branch, warehouse, commissary, Head Office, project site, or future temporary site.                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| Scope                                 | The company, brand, location, department, or project access assigned to a user.                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| Food Cost Analysis                    | The authorized source workspace for comparing posted sales, theoretical recipe cost, and available inventory-ledger evidence for a selected branch and business date. While `DEC-0062` definitions remain under review, Food Cost values are deliberately withheld from Overview and no new Food Cost exception reminders are generated; this does not mean zero exceptions or unavailable source data. |
| Today’s work                          | The read-only, server-derived priority view at the top of the Operations Dashboard. It shows bounded assigned approvals and operational exceptions for the selected authorized scope, including location, owner, timing, severity, status, and next action. Opening an item still rechecks permission and workflow rules in the source workspace. |
| My Tasks                              | The server-owned, paginated operational action queue in the Overview navigation. It combines personally attributable work and role-pooled work that the user may act on in the selected scope. Current enrollment includes the requester's own draft Purchase Request submissions; eligible Purchase Order, Transfer, Wastage, Stock Adjustment, and draft Receiving Report actions; assigned first-pass Stock Count start, entry, or submission; independent Branch Operations and Food Safety review; pooled returned-record correction for those two sources; risk-controlled Incident resolution; and risk-controlled Maintenance completion. It opens the authoritative source record and rechecks permission, actor, status, and scope rules; it is not Approval Inbox, project My Work, or a grant of action authority. |
| Assigned Stock Count work             | First-pass physical-count work belonging to the recorded counter. Depending on current state and permission, My Tasks shows exactly one Start, Enter, or Submit action. It excludes future-scheduled starts, empty snapshots, recount, review, cancellation, and variance-adjustment work. |
| Role-pooled work                      | A current action available to any user whose role and selected scope satisfy the source workflow, rather than a task assigned to one named person. Another authorized user may complete it first, so opening the source record rechecks status and authority. Returned Branch Operations checklist correction, Food Safety log correction, eligible Incident resolution, and eligible Maintenance completion are currently role-pooled. |
| My Tasks priority order               | The stable queue order: critical, high, medium, then low priority; within a priority, dated work by earliest absolute due date before undated work; then oldest creation time, source order, and record ID. Existing sources without native priority/due dates use high priority and no due date. Overdue/today wording is calculated in the operating timezone for display and does not rewrite cursor order. |
| Dashboard drilldown                   | A closed, server-owned link from a dashboard card to a matching authorized source-list profile. It does not carry scope IDs, record authority, or arbitrary filters. The current `Open POs`, `Open PRs`, `Transfer Follow-up`, `Receiving Follow-up`, and `Ledger Variance` drilldowns reuse their selected-location membership across counts, pages, and CSV export. |
| Dashboard source status               | The availability and observation details for each dashboard source the server authorized and attempted for the selected scope. `Dashboard assembled` and `Checked` are response-observation times displayed in `Asia/Manila`, not proof that records are fresh or complete. `Source data as of` appears only when a source provides its own documented as-of time. An unavailable source makes the affected dashboard view partial rather than zero; opening the source workspace rechecks current access and shows the authoritative record. |
| Receiving Follow-up                   | A read-only, selected-location dashboard profile for unposted drafts, receipts being posted, active recorded discrepancies, and discrepancy-bearing reversals in progress. Each row states why it is included. The profile does not resolve discrepancies or grant posting, reversal, Purchase Order, or inventory authority; open the Receiving Report for current facts and independently authorized actions. |
| Ledger Variance                       | A diagnostic difference between a cached stock balance and the sum of immutable inventory movements for the same selected-location storage, item, and normalized lot/expiry key. It is calculated as cached balance minus ledger total to six decimals. It does not authorize a balance edit or Stock Adjustment; investigate the exact ledger trace and use the controlled source workflow only when a real operational correction is justified. |
| Tenant-wide role administration       | The high-risk access control labeled `Administer tenant-wide roles` (`core.tenant_role_administer`). It is required to view or change the tenant-owned role catalog and tenant-wide user role assignments. Core Administration access or company Manage scope alone is not enough.                                                                                                                                                                                                                                                   |
| Tenant-wide role assignment           | A role assignment that is not limited to the company selected when it was granted. The user's active scopes still determine which company and location records the user can access. The role does not grant company-record access by itself.                                                                                                                                                                                                                                                                                         |
| Selected-company target eligibility   | The rule that a role grant, deactivation, request, or review may concern only an active user with a current active company assignment for the selected company or an active location assignment to an active location in that company. A default company alone does not qualify.                                                                                                                                                                                                                                                     |
| Controlled role revocation            | Removal of an active role by an authorized administrator with a recorded reason, audit event, privilege-epoch change, and session invalidation. A sensitive role requires controlled approval when granted, but that grant classification must not prevent its later revocation.                                                                                                                                                                                                                                                     |
| Initial-role onboarding               | Creating a user with an initial role. This requires an initial active location in the selected company, and the system creates the location membership and role assignment together.                                                                                                                                                                                                                                                                                                                                                 |
| Location-primary workforce scope      | Workforce access for a location-bound record requires an active assignment to that exact location. A shared brand does not extend access to another location. The record's brand must match the location's brand, and any department must be active in the same company; a department assignment does not currently narrow otherwise valid workforce location access.                                                                                                                                                                |
| Source-record access                  | The current permission and scope required to view the business record that an attachment, export, notification, or project link belongs to. Having a copied link or attachment ID does not grant access.                                                                                                                                                                                                                                                                                                                             |
| Canonical evidence mutation authority | The source-specific authority the ERP derives before it links, uploads, adds metadata to, or archives controlled evidence. This is normally the source write permission; a project-requirement owner may instead use the controlled owner path only after live project-view, ownership, and project-scope checks pass. A caller-supplied permission name cannot weaken or replace the canonical policy. For example, period-close evidence changes require `finance.period_close.manage`; Core Administration alone is insufficient. |
| Controlled evidence                   | A private file linked to an authorized business record and managed through upload verification, malware scanning, exact-version integrity checks, retention controls, and audit history. Uploading controlled evidence never approves or changes the source record by itself.                                                                                                                                                                                                                                                        |
| Controlled evidence qualification     | A future action-time record proving which exact clean, available, durable evidence versions satisfied an approved action policy. The technical foundation is currently dormant: no approved policy, real action adapter, selector, or user-facing qualification behavior is active. An upload, filename, URL, or free-text evidence reference is not qualification. |
| Evidence quarantine                   | The protected state used while an uploaded file is incomplete, awaiting safety checks, or unable to pass them. Quarantined evidence cannot be downloaded or used as available evidence.                                                                                                                                                                                                                                                                                                                                              |
| Required evidence                     | Controlled evidence that the server associates with a specific workflow action. Its active link cannot be archived through the normal workspace action. Whether it is sufficient for a high-risk finance action still follows the approved evidence matrix; arbitrary text is not verified evidence.                                                                                                                                                                                                                                 |
| Project requirement evidence          | A controlled file used to satisfy a project-level or task-bound evidence requirement. Project-level evidence does not need an artificial task; task-bound evidence is linked only to the requirement's own task. Uploading the file does not submit, approve, waive, or otherwise decide the requirement.                                                                                                                                                                                                                         |
| Legal hold                            | A preservation control placed by an authorized, MFA-assured user. It prevents normal archival or disposal of the evidence. Hold release and physical purge are not available in the current release.                                                                                                                                                                                                                                                                                                                                 |
| Verified external reference           | A future structured reference whose identity, verifier, verification time, scope, and validity are controlled by an approved policy. A filename, URL, voucher number, or free-text evidence note is not a verified external reference.                                                                                                                                                                                                                                                                                               |
| Restricted project                    | A project visible only through active project membership, an exact project scope, or authorized project management with company-level manage scope. Broad branch, brand, or department scope alone does not grant access.                                                                                                                                                                                                                                                                                                            |
| Purchase Request (PR)                 | A request to obtain goods or services that may require approval before purchasing.                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| Purchase Order (PO)                   | An approved order issued to a supplier.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| Receiving Report                      | The record of goods accepted, rejected, damaged, short, or partially delivered.                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| Retry-safe receipt creation           | Goods Receipt draft creation protected by a server-checked idempotency key and canonical request hash. An exact retry returns the same authorized draft; changed data or scope requires a new key and is rejected if the old key is reused.                                                                                                                                                                                                                                                                                     |
| Transfer                              | A controlled movement of stock from one location to another.                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| Stock Count                           | A physical count of inventory compared against system stock. In a blind count, counters do not receive system quantity, calculated variance, reviewer notes, or variance-disclosing audit details; only an authorized reviewer in the assigned scope may view variance information. A count does not directly change stock. |
| Count Variance                        | The difference between a physical count and the system quantity at the count cutoff. It is reviewer-only information for blind counts. Reviewing a variance does not correct stock: any permitted correction must use the controlled Stock Adjustment workflow and separate posting. |
| Wastage                               | Stock lost through spoilage, damage, expiry, preparation loss, or another approved reason.                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| Stock Adjustment                      | A controlled inventory correction that requires a reason and may require approval.                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| Approval                              | A recorded decision that allows, rejects, or returns a request under the configured approval rules.                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| Approval Step                         | One assigned decision point in a configured approval route. Approving a step advances the record to the next eligible approver when another step remains; the source record stays pending approval until the final step is approved.                                                                                                                                                                                                                                                                                                 |
| Canonical approval decision           | The single server-controlled approval action that rechecks the active step, live authority, scope, segregation rules, and source state, then commits the approval outcome and any permitted final source effect together. Normalized canonical routing remains disabled until its documented cutover gates pass. |
| Current approval amount               | The typed petty-cash amount intended to be carried by an active normalized approval chain. It is separate from the final approved amount. Approval-time amount changes are blocked in both routing modes. The normalized route's per-step intent writer and proposal-version compare-and-set control are implemented behind the disabled routing flag, but PostgreSQL behavioral acceptance and production activation remain pending. |
| Approval step intent                  | The immutable typed petty-cash record of the unchanged amount and proposal version carried by one acted approval step, together with its actor and approval lineage. Its writer is implemented only for feature-disabled normalized routing and is not currently available to users. It preserves decision history but grants no permission to change an amount and creates no payment, bank, journal, or Petty Cash ledger effect. PostgreSQL behavioral acceptance and routing activation remain pending. |
| Role-scoped approval                  | An approval step assigned to an approval role instead of a fixed person. The ERP checks the user's current role, permission, effective assignment, scope, active status, and segregation rules when showing and acting on the work. Role-scoped activation creates no personal notifications and the live Approval Inbox is authoritative; a direct-user step may create one personal notification for the persisted assignee, but receiving it is not authority to approve. PostgreSQL parity and source-concurrency gates still apply before normalized routing is enabled. |
| Approval routing snapshot             | The protected company, permission, assignment, scope-group, prohibited-actor, activation, and due-date context recorded for an approval step. Once the normalized version is active, this context is not rewritten; later authority changes are checked live and may remove the item or deny a stale action. |
| Budget Revision pre-review state      | A protected normalized-routing state in which a submitted Budget Revision has an active approval instance but every route step is waiting. An authorized commitment-fit review may make only the first step actionable; an authorized cancellation may terminate the complete waiting route. This remains feature-disabled and is not yet a user-facing production procedure. |
| Stale approval action                 | An approval, review, or cancellation request whose protected source record or approval route changed while the request was waiting. The ERP rejects it instead of applying it to the newer state. |
| Audit Trail                           | The history of who created, changed, approved, rejected, cancelled, or otherwise acted on a record. Protected audit history is append-only: corrections add another event instead of changing or deleting the original event.                                                                                                                                                                                                                                                                                                         |
| Append-only history                   | Protected history whose existing entries cannot be edited, deleted, or cleared in bulk. In the current implementation this protection applies to `AuditEvent`, `ProjectActivityEvent`, and `InventoryMovement`. Corrections use an authorized correction or reversal workflow that adds new history and preserves the original entry. Production activation remains subject to the release gate.                                                                                                                                    |
| AuditEvent                            | The protected append-only record of an important ERP action or decision. A correction creates another audit event; it does not rewrite the original event.                                                                                                                                                                                                                                                                                                                                                                            |
| ProjectActivityEvent                  | The protected append-only activity record for a project or task, such as status, assignment, comment, checklist, evidence, risk, or milestone activity where implemented. A correction creates new activity instead of deleting or changing the earlier entry.                                                                                                                                                                                                                                                                         |
| InventoryMovement                     | The protected append-only inventory-ledger entry created by a posted stock workflow. A correction uses an authorized linked reversal or other controlled workflow; it never edits or removes the original movement.                                                                                                                                                                                                                                                                                                                   |
| Scope context                         | The company, optional brand, and location selected for a workspace. An empty brand selection is not a brand identifier; the server applies the applicable company/location scope before reading or posting operational records. |
| Runtime MFA                           | A live multifactor check performed by OGFI ERP using an enrolled authenticator or an unused recovery code. External MFA evidence alone does not satisfy it.                                                                                                                                                                                                                                                                                                                                                                          |
| MFA challenge lock                    | Automatic revocation of an incomplete MFA challenge after the configured failed-attempt limit. A new password sign-in is required before trying again.                                                                                                                                                                                                                                                                                                                                                                               |
| Authentication protection window      | A short server-controlled period used to limit repeated sign-in or MFA attempts. Under heavy or suspicious traffic, the ERP may temporarily reject another attempt without revealing whether an organization code, email, or account exists. |
| MFA step-up                           | A fresh runtime MFA check required before a sensitive action. The current assurance window is 15 minutes.                                                                                                                                                                                                                                                                                                                                                                                                                            |
| Recovery code                         | One of 10 single-use codes issued after MFA enrollment. It can replace an authenticator code when the device is unavailable.                                                                                                                                                                                                                                                                                                                                                                                                         |
| Application session                   | A server-controlled signed-in session that can expire or be revoked without deleting the user account.                                                                                                                                                                                                                                                                                                                                                                                                                               |
| Privilege epoch                       | A server-maintained access version for a user. Role, role-permission, or scope changes refresh it so an older session must revalidate before protected access continues.                                                                                                                                                                                                                                                                                                                                                             |
| Activation link                       | A single-use link sent directly to the account email address so a user can create or replace local credentials. It expires after 30 minutes and is not displayed to administrators.                                                                                                                                                                                                                                                                                                                                                  |
| Controlled recovery                   | The audited process for recovering an existing account: one administrator submits identity evidence and a different MFA-assured administrator approves or rejects the request.                                                                                                                                                                                                                                                                                                                                                       |
| First-administrator bootstrap         | A deployment-only, one-time tenant ceremony that issues the initial administrator activation through a restricted file. It cannot be reused for recovery or another user.                                                                                                                                                                                                                                                                                                                                                            |
