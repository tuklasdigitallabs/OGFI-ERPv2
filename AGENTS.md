# AGENTS.md — OGFI ERP Project Rules

## 1. Project identity

OGFI ERP is a tenant-ready, multi-company, multi-brand, multi-branch ERP for F&B and restaurant operations.

The system must support Company → Brand → Location, where a location can be a restaurant branch, Head Office, warehouse, commissary / central kitchen, project site, or future temporary site.

**Current delivery scope:** Phase I — procurement, approvals, suppliers, receiving, inventory, transfers, stock counts, wastage, stock adjustments, reporting, notifications, audit trails, and role-based access. **Phase 1.5 is specified next:** Projects & Implementation Tracker, delivered after Phase I stabilization unless the user explicitly changes release priority.

Do not build generic ERP features at the expense of restaurant operational controls.

## 4. Core business and data rules

1. Every tenant-owned record includes `tenant_id`; every company-owned operational record includes `company_id`.
2. Transaction records must carry their required scope: company, brand where applicable, location, department or cost center where applicable, requester, owner, dates, status, and audit history.
3. Branch users can access only assigned scopes. Management access comes from role plus explicit scope assignment; never from UI hiding alone.
4. Head Office, Main Warehouse, and Commissary are company-level locations and can serve more than one brand.
5. Important records are never hard-deleted. Use cancellation, reversal, archival, or deactivation while preserving history.
6. Policy values such as approval thresholds, approvers, escalation intervals, stock-count frequency, and attachment requirements must be configurable. Do not hardcode them into page components or services.
7. Store timestamps in UTC. Render dates and times in the company or user timezone; OGFI’s initial operational default is `Asia/Manila`.
8. Use database-generated IDs or UUID/ULID identifiers. Human-readable document numbers must be separate, unique, and generated by an approved service.

## 5. Procurement and inventory controls

These controls are non-negotiable.

- Low stock must **not** automatically create a Purchase Order.
- The system must first check the assigned Main Warehouse or warehouse availability.
- When warehouse stock is available, create a **Transfer Request**.
- When warehouse stock is unavailable, create a **Purchase Request**.
- Purchasing flow: `Purchase Request → Approval → Quotation Comparison when required → Purchase Order → Receiving → Inventory Ledger Update`.
- Transfer flow: `Transfer Request → Approval when required → Dispatch → Receiving Confirmation → Inventory Ledger Update`.
- Receiving must track ordered, delivered, accepted, rejected, damaged, short, and outstanding quantities.
- A partial receipt keeps the PO open for the outstanding quantity unless an authorized close or cancellation action is posted.
- Inventory balances are derived from an immutable inventory movement ledger. Never update a stock balance directly without a corresponding posted movement.
- Transfer dispatch and receipt must not duplicate inventory. Use idempotent, transactional posting.
- Wastage, count variance, and stock adjustments require a reason, evidence when configured, required approval, and audit history.
- No user may approve their own money-related request or bypass segregation-of-duties controls.

## 6. Architecture and coding rules

### Baseline architecture

- Build a modular monolith first.
- Use the approved stack in `docs/core/05-technical/TECH_STACK_DECISION.md`.
- Keep UI, route handlers, domain services, validation, authorization, persistence, and background jobs separated.
- Browser components must not perform privileged writes directly to the database.
- Route handlers and server actions must delegate controlled business actions to domain services.
- Apply tenant and scope authorization in the service/data-access layer, not only in the interface.

### Database and migrations

- Use PostgreSQL and Prisma according to the data model and migration rules.
- Never edit a production database manually to change application schema.
- Every schema change requires a reviewed migration, rollback considerations, and data dictionary update when business fields change.
- Wrap multi-record workflow posting in database transactions.
- Use explicit locking or concurrency checks for inventory posting, count closure, and document-number allocation.

### Validation and errors

- Validate all write input at the API boundary using shared schemas.
- Return stable, user-safe error codes and field-level validation messages.
- Do not expose database errors, stack traces, secrets, or internal authorization details to users.
- Use idempotency keys or equivalent protection for retried posting endpoints.

### Dependencies

- Prefer existing project dependencies and patterns.
- Do not add a dependency merely to avoid small, clear implementation work.
- Explain any significant new dependency in the completion report.
- Do not introduce a second UI system, ORM, authentication model, or job queue without an approved technical decision.

## 7. UI and UX rules

Follow all documents under `docs/core/04-design/`, especially:

- `UI_IMPLEMENTATION_STANDARD.md`
- `DESIGN_TOKENS.md`
- `COMPONENT_LIBRARY.md`
- `DASHBOARD_RULES.md`
- `MOBILE_RULES.md`
- `UX_STATES_AND_VALIDATION_SPEC.md`

### Official direction

Use a Modern SaaS interface: clean structure, restrained blue accents, white cards, balanced tables, clear status pills, role-aware dashboards, and responsive task-focused mobile workflows.

UI/UX readiness is part of implementation completion. A workflow is not complete merely because the schema, services, routes, and tests exist. Client-facing workspaces must also meet the approved layout, navigation, readability, and task-efficiency standards before the milestone may be called implemented or ready for UAT, unless the user explicitly authorizes a clearly labeled backend-only foundation slice.

### Visible workflow completion gate

Do not call a workspace, workflow, phase slice, builder, tracker, dashboard, or administrative tool complete while any visible tab, section, panel, action area, or navigation destination is only a passive placeholder or partially wired experience.

Every visible surface must satisfy one of these outcomes before completion is claimed:

- It has the working create, edit, remove, submit, approve, post, export, configure, or workflow actions that the UI label implies.
- It is intentionally read-only, and the screen clearly explains why it is read-only and where the authoritative action happens.
- It is disabled because of status, role, scope, or policy, and the disabled state tells the user the reason.
- It is explicitly labeled as a preview, future module, or backend-only foundation that the user has approved for the current slice.

Never use vague completion thresholds such as "real enough", "mostly wired", "displaying data", "backend complete", or "good enough for now" to mark implementation work complete. If a page calls itself a builder, setup center, control center, inbox, queue, dashboard, register, or workspace, its visible tabs and sections must behave like that product concept implies. If a tab is shown, it must either provide useful working controls or have a deliberate documented read-only purpose.

Before reporting a feature as done, perform a visible-surface checklist: list the tabs/sections/actions introduced or affected, confirm each has a working behavior or intentional read-only state, and verify the relevant tests/typechecks. If any visible surface is incomplete, report it as pending instead of complete.

### Non-negotiable UI behavior

- Use one shared spacing token for gaps and padding.
- Do not sacrifice transaction clarity for visual minimalism.
- Do not build or extend long stacked-card workspaces for operational modules. Multi-workflow areas must use a clear workspace model: route/subworkspace tabs, filters/search, paginated list or queue, selected-record detail/drawer, and contextual actions.
- Cards are allowed for compact KPI strips, repeated record summaries on mobile, modals, and genuinely framed tools. They must not become the primary layout for every finance, workforce, procurement, inventory, or operations function.
- Lists and queues with more than 10 records must provide pagination or an approved virtualized/incremental loading pattern. Do not rely on `slice(0, 10)` or unpaginated browser-only filtering for operational records.
- Do not repeat large inline action forms in every row or card. Record-specific workflow actions belong in a selected-record drawer, detail page, or focused action composer.
- Use modals only for short single-purpose forms. Long transaction entry, multi-line entry, evidence-heavy actions, and review workflows must use a drawer, sheet, stepper, full-page task mode, or equivalent focused workspace.
- Important screens must visibly show company, brand where relevant, location, requester/owner, status, dates, next action/current approver, and audit/activity access.
- `All Brands / All Locations` is only for authorized dashboards and reports, never for stock posting, receiving, or location-specific transactions.
- Use clear action labels such as `Create Purchase Request`, `Request Stock`, `Receive Delivery`, `Dispatch Transfer`, and `Log Wastage`; avoid ambiguous universal actions.
- Data tables must support the documented filters, search, export, status visibility, empty/error/loading states, and mobile card behavior.
- Design and validate desktop, tablet, and mobile behavior. Mobile branch workflows are task-first, not reduced desktop screens.

## 8. Engineering workflow and working style

These rules apply to the parent agent and every subagent.

### Before editing

For non-trivial work:

1. Read only the relevant code, document sections, and immediate dependencies.
2. Identify the existing pattern in the affected area.
3. State a short plan and the files expected to change.
4. State only assumptions that materially affect the implementation.
5. Define a verifiable success condition.
6. Proceed when the scope is clear. If a material ambiguity could change the implementation, stop and ask the user before editing.

For small, unambiguous changes, proceed directly without a planning ceremony.

### During editing

- Make the smallest correct change that satisfies the request.
- Do not refactor, rename, move, reformat, or improve unrelated code.
- Treat each task as self-contained. Do not re-open or re-read earlier context unless it is needed to verify current behavior.
- Do not replace working logic with placeholders.
- Do not continue adding functional sections to a workspace after it becomes a long stacked-card or long-scroll page. Stop and restructure the information architecture first, or ask the user for approval to proceed as a backend-only foundation.
- Do not stop an implementation pass after making a surface "display data" if the user asked to complete the workflow. Finish the visible actions implied by the page, or explicitly mark the remaining visible surfaces as pending blockers before stopping.
- Treat relevant Critical or High findings in `docs/core/04-design/UI_UX_WORKSPACE_AUDIT.md` as implementation gates for new work in that workspace. New work must remediate or avoid the cited pattern, not deepen it.
- Preserve backward compatibility unless the task explicitly allows a breaking change.
- Keep changes coherent, reviewable, and traceable to the request.
- Default to medium reasoning effort. Escalate only for genuinely hard, high-risk, multi-file, financial, inventory, security, data-integrity, or architecture work.

### Shell and tool discipline

- Keep shell usage quiet and targeted.
- Prefer focused commands such as `git status --porcelain`, `git diff --stat`, `git diff -- <path>`, `rg`, `grep`, and `tail` scoped to relevant paths or lines.
- Do not dump full logs, full build output, full generated files, or broad repository listings into context.
- Surface only the relevant error lines, test failures, or code snippets needed to diagnose the issue.
- Ignore `node_modules`, build output, generated code, and large data files unless directly relevant.

### Long-session discipline

When earlier context is stale, conflicting, or too large to safely carry forward, suggest a compact handoff or a fresh session rather than relying on memory or re-reading the entire repository.

### Validation

After editing, run the relevant scripts once scaffolded:

```text
pnpm lint
pnpm typecheck
pnpm test
pnpm test:e2e
```

Run only the relevant subset where the project provides targeted commands, but never claim success without actual verification.

## 13. Subagent deliberation and decision protocol

For **material decisions** involving workflow, permissions, approvals, inventory, money, database schema, tenancy, integrations, deployment, reporting definitions, release readiness, or user adoption, use the parent-led deliberation process before implementation.

A material decision is any decision that is difficult to reverse, changes source-of-truth data, changes user authority, affects inventory or money, changes a public API or schema, or materially affects branch operations.

Read and follow:

- `docs/core/00-governance/SUBAGENT_DELIBERATION_PROTOCOL.md`
- `docs/core/00-governance/DECISION_SCORECARD.md`
- `docs/core/00-governance/DECISION_RECORD_TEMPLATE.md`

### Parent-agent responsibilities

The parent agent is the **Decision Chair**. It owns the task end-to-end and must:

1. Create a concise decision brief before delegating.
2. Select only the relevant specialists; use no more than four first-round agents unless explicitly justified.
3. Ask first-round agents for independent analysis before they see other positions.
4. Compile a debate packet and send targeted challenge questions only where a real disagreement or material risk exists.
5. Apply hard gates before scoring options.
6. Do not use majority vote. A credible inventory, security, authorization, audit, or data-integrity blocker cannot be overridden by preference or convenience.
7. Record unresolved policy questions in `docs/core/00-governance/OPEN_DECISIONS_AND_ASSUMPTIONS.md`.
8. Ask Mithi to create a formal decision record only after the parent agent or authorized human confirms the conclusion.

### Subagent model fallback

Use the default inherited subagent model first. If a subagent fails because
`GPT-5.3-Codex-Spark` or the current default subagent model is at its usage limit,
do not repeatedly retry the same limited model. Close the failed subagent thread and
continue with the smallest necessary council using an explicit fallback:

- Use `gpt-5.4` for material product, architecture, data, security, workflow,
  permissions, approval, inventory, finance, integration, release, or UI/UX
  decisions.
- Use `gpt-5.4-mini` for lightweight checklist review, copy review, focused
  verification, or low-risk sidecar analysis.

Use no more fallback subagents than the decision actually needs, and close each
thread when its result is no longer needed.

### Required first-round output

Every deliberating subagent must provide:

- Position and recommended option
- Alternatives considered
- Why the recommendation works
- Likely failure modes
- Assumptions and evidence
- Hard blockers or non-negotiable controls
- Required safeguards or tests
- Confidence level: High / Medium / Low

### Required challenge-round output

When given a debate packet, each challenger must state:

1. The strongest part of the proposed recommendation.
2. The most likely failure mode.
3. Evidence that would disprove or weaken it.
4. The safeguard needed to make it acceptable.
5. Severity: Blocking / Serious but manageable / Minor.

### Hard gates

An option cannot proceed unless it preserves all applicable controls:

- Tenant, company, brand, and location scope isolation
- Server-enforced authorization
- Approval segregation and no self-approval where required
- Immutable inventory-ledger and audit-trail principles
- Transactional consistency and idempotency where required
- Phase scope discipline
- Recovery, reversal, or rollback path appropriate to the change

### Implementation lock

No write-capable implementation agent may implement a material decision while its status is `OPEN`, except for a clearly labeled non-production prototype explicitly authorized by the parent agent.

### Sandbox and authority reminder

Read-only instructions are mandatory behavioral constraints, not merely a sandbox setting. Parent sessions can have broader permissions than child defaults. No reviewer may edit code, production configuration, live data, or source-of-truth policy documents unless the parent explicitly authorizes the exact write scope.


## 14. Projects & Implementation Tracker — Phase 1.5

The Projects & Implementation Tracker is an ERP-native coordination module. It is a **linked work-management layer**, not a generic standalone Trello clone and not a replacement for controlled ERP records.

### Phase 1.5 purpose

Use it to coordinate implementation work such as ERP rollout, IT implementation, operational improvement, renovation, maintenance, marketing activation, training rollout, audit corrective action, supplier onboarding, compliance, and future branch-opening work.

### Non-negotiable controls

- Every project belongs to a company and has a defined visibility scope. It may also be linked to a brand, location, department, or project site.
- Task cards may **link to** Purchase Requests, Purchase Orders, Receiving Reports, Transfers, Suppliers, Inventory records, approvals, or future Expansion records. They must not directly alter the status, financial effect, approval state, or inventory state of those source records.
- The linked ERP record remains the source of truth for money, inventory, supplier commitments, approvals, and controlled status changes.
- Project and task access must enforce company, project membership, role, and assigned scope in services/data access. UI hiding alone is never authorization.
- Sensitive projects require explicit membership or project scope. A user with broad department access does not automatically receive access to restricted projects.
- A task marked `Blocked` requires a blocker reason. A completed task requires a completed-by user and timestamp.
- Project activity, task status changes, assignments, comments, attachments, checklist changes, links, risks, and milestones are auditable. Important activity is never hard-deleted.
- Do not build advanced Gantt calculations, workload balancing, external contractor portals, public links, or custom automation builders in Phase 1.5.

### Required source documents

Before editing this module, read:
- `docs/phases/phase-01-5-projects-implementation/README.md`
- `docs/phases/phase-01-5-projects-implementation/PROJECTS_IMPLEMENTATION_PRODUCT_SPEC.md`
- relevant workflow and UI specifications in the same phase folder
- `docs/core/02-controls/ERP_ROLES_AND_PERMISSIONS.md`
- `docs/core/02-controls/SECURITY_AND_AUDIT_MODEL.md`
- `docs/core/03-data/ERP_DATA_DICTIONARY.md`
- `docs/core/04-design/UI_IMPLEMENTATION_STANDARD.md`

### Delivery sequence

1. Foundation models, project scope, permissions, templates, and audit activity.
2. Projects, members, board/list view, task cards, comments, attachments, and task activity.
3. Checklists, milestones, risks/blockers, notifications, and linked ERP records.
4. Calendar and reporting only after core task-state integrity and mobile task completion are verified.

Use structured deliberation before any decision that changes project visibility, source-record linking, status semantics, attachment access, notification escalation, or future Expansion integration.
