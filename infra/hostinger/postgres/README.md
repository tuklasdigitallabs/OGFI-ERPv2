# Hostinger PostgreSQL role contract

These files implement the database-role portion of `DEC-0049` without selecting how PostgreSQL is packaged. They apply to PostgreSQL on the same Hostinger VPS whether the approved activation later uses a host service or a dedicated production container. They do not install PostgreSQL, open ports, alter DNS, or deploy services.

`bootstrap-roles.sql` is a superuser/cluster-administrator ceremony. It creates environment-qualified roles without passwords, clears every owner/migrator/runtime membership edge, restores exactly migrator → owner with `ADMIN FALSE`, `INHERIT FALSE`, and `SET TRUE`, adopts the exact target database, `public` schema, and supported non-extension public objects from a legacy owner or `--no-owner` restore executor, and makes new migrator sessions assume the non-login owner. This adoption must run before the restored database can accept application traffic. Provision the migrator and runtime SCRAM passwords separately without placing them in SQL, shell history, source control, or logs.

The same SQL is reusable by the disposable database lifecycle with `contract_scope=disposable`, `app_environment=test`, a positively identified `ogfi_test_*` or `ogfi_demo_disposable_*` database, and one per-run prefix shaped as `ogfi_test_<token>_<nonce>_{owner,migrator,runtime}`. The admin parent runs bootstrap, then connects as the migrator. Bootstrap sets the database-specific default `role` to the non-login owner, so every new Prisma migrator connection proves `session_user=<per-run migrator>` and `current_user=<per-run owner>` without a URL `options` parameter. The child application receives only the runtime URL. Hosted calls use `contract_scope=hosted` and retain the exact `ogfi_prod_*` / `ogfi_stg_*` names and root-owned credential-file controls.

`reconcile-ownership-and-grants.sql` runs through the controlled migrator after Prisma migration. It removes PUBLIC table/sequence/routine access, removes runtime routine and protected-column ACLs, restores exact runtime table/sequence defaults, and revokes default function execution. The runtime receives ordinary application CRUD but exactly `SELECT` and `INSERT` on `AuditEvent`, `ProjectActivityEvent`, and `InventoryMovement`, and no access to `_prisma_migrations`.

`verify-role-contract.sql` fails on an unexpected schema, unsupported or wrongly owned public object, unsafe default ACL, protected column ACL, unexpected role-membership path, callable non-extension public routine, or callable `SECURITY DEFINER` routine in a non-system schema. Hosted staging/production have no callable-definer allowlist. Disposable databases allow only `ogfi_disposable_control.verify_database_identity()` with its exact zero-argument signature, owner, SQL/STABLE mode, fixed `pg_catalog` search path, bounded runtime execute grant, and zero runtime marker-table or column privileges.

`adversarial-role-drift.sql` is a disposable PostgreSQL 17 fixture for proving these failures. The lifecycle invokes all eight cases only inside `authorization-all`, using a per-run `ogfi_adv_<run>_<128-bit nonce>` role bound to the disposable database identity. Each expected verifier failure is repaired through bootstrap, reconciliation, or explicit administrator cleanup, then owner/runtime verification and double cleanup must pass. The fixture rejects non-disposable targets and must never run against staging or production.

The application environment contains only the runtime URL. The migrator URL is a root-owned systemd credential consumed by `scripts/db-migrate-controlled.mjs`; it is never mounted into the web, evidence broker, Caddy, or a future worker. Run `verify-role-contract.sql` from both the migrator/owner and runtime sessions and retain only sanitized fingerprints and PASS markers.

Restoring a portable `--no-owner --no-privileges` dump does not restore this boundary. Keep the restored environment in maintenance mode, rerun bootstrap and ownership/grant reconciliation, rerun the complete append-only contract, and only then allow application traffic.

Production remains **NO-GO** until the external PostgreSQL packaging choice, private connection boundary, credential custodians, backup retention/RPO/RTO, and hosted exact-SHA role/guard evidence are approved.
